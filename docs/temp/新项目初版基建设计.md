# 新项目初版基建设计

> 本文只讨论新项目的基建结构，不讨论 NL2SQL 业务节点、QueryPlan 细节、评测集设计或具体 SQL 生成质量优化。
>
> 目标：在引入 LangGraph 的前提下，先把配置、日志、观测、LLM、数据库、向量库、Graph runtime、依赖装配这些底座边界定清楚，避免旧项目里 token 统计、日志、运行记录和外部依赖接线散落到各处。

## 1. 核心判断

新项目底座建议按下面几类组织：

```text
platform      项目运行设施
integrations  外部系统适配
bootstrap     依赖装配
workflows     LangGraph 工作流运行层
interfaces    CLI / API 入口
```

其中：

```text
platform
  处理配置、日志、观测、持久化、路径、统一错误等项目级运行设施。

integrations
  处理 LLM、数据库、向量库、embedding、tracing provider 等外部系统。

bootstrap
  负责把 config、logger、observability、integrations、services、workflows 接起来。

workflows
  放 LangGraph graph、state、nodes、edges，以及 graph runtime wrapper。

interfaces
  放 CLI / HTTP / 未来 UI 或 MCP 入口。
```

这套结构的目的不是“目录好看”，而是让下面几件事不要再次混在一起：

```text
业务流程
外部依赖
token 统计
日志
运行记录
checkpoint
CLI 输出
```

## 2. LLM 放 integrations 是否合适

结论：**LLM 放到 `integrations/llm` 比旧项目更好。**

原因是 LLM 本质是外部系统，和数据库、向量库、embedding 一样，需要统一适配。

`integrations/llm` 应负责：

```text
1. 创建模型客户端。
2. 读取 provider / model / temperature 等配置。
3. 统一 LangChain callbacks。
4. 统一 LangSmith / tracing metadata。
5. 统一 token usage 采集入口。
6. 统一模型调用错误包装。
7. 提供 chain 层的基础调用封装。
```

业务服务和 LangGraph node 不应该直接：

```text
new OpenAI client
new ChatModel
设置 callbacks
设置 LangSmith metadata
手动解析 provider usage
写 token summary 文件
```

不过要注意一个重要边界：

```text
prompt payload builder 不应该放 integrations/llm。
```

推荐边界：

```text
services/nl2sql/prompt_payload_builder.py
  负责把业务输入整理成 prompt 变量。

integrations/llm/chains/nl2sql_chain.py
  负责加载 prompt 模板、调用 LangChain model、返回模型输出。
```

也就是说：

```text
业务语义怎么变成 prompt 变量 -> services
怎么调用模型 -> integrations
```

如果把 prompt 业务逻辑也塞进 `integrations/llm`，那 `integrations` 会开始理解业务，边界就会变差。

## 4. platform 层设计

`platform` 是项目运行设施层。

它不承载 NL2SQL 业务语义，也不直接访问数据库、向量库、LLM。

### 4.1 platform/config

职责：

```text
1. 读取 YAML / env 配置。
2. 解析为稳定配置模型。
3. 提供 AppConfig。
```

建议配置模型：

```text
AppConfig
ModelConfig
EmbeddingConfig
DatabaseConfig
VectorStoreConfig
WorkflowConfig
ObservabilityConfig
LoggingConfig
```

不做：

```text
不创建模型客户端。
不创建数据库连接。
不创建 vectorstore。
不决定 graph 分支。
```

### 4.2 platform/logging

职责：

```text
1. Python logging 基础配置。
2. 日志格式。
3. console/file handler。
4. logger 命名规范。
5. 日志目录初始化。
```

不做：

```text
不统计 token。
不写业务 run record。
不解析 graph state。
```

### 4.3 platform/observability

这是旧项目最容易散的一块，新项目必须收拢。

建议文件：

```text
trace_context.py
  负责 request_id / run_id / thread_id / trace_id 的生成和规范化。

usage_tracker.py
  包装 LangChain usage callback 或 LangSmith usage 信息。

run_recorder.py
  写业务运行记录，例如 command、question、status、sql、row_count、error。

langsmith.py
  LangSmith 开关、project、tags、metadata 的配置辅助。
```

旧项目里类似下面的能力，应该集中迁移到这里：

```text
reset_current_run_token_usage
flush_token_usage_summary
write_token_summary_json
write_run_record
chain_log_paths
```

新项目不要让 CLI、chain、node 各自手写 token 和 run record。

### 4.4 platform/persistence

职责：

```text
1. 创建 LangGraph checkpointer。
2. 根据配置选择 memory / sqlite / postgres 等 provider。
3. 隐藏 LangGraph persistence 细节。
```

第一版可以只支持：

```text
memory checkpointer
```

后续再加：

```text
sqlite checkpointer
postgres checkpointer
```

### 4.5 platform/errors

建议定义项目级异常基类：

```text
SQLAgentError
ConfigurationError
IntegrationError
WorkflowError
PolicyViolationError
```

不要让外部库异常直接穿透到 CLI / API 层。

## 5. integrations 层设计

`integrations` 是外部系统适配层。

它只回答：

```text
怎么连接外部系统？
怎么把外部系统结果转成项目可理解的结果？
```

它不回答：

```text
NL2SQL 下一步该走哪里？
是否 retry？
是否需要澄清用户？
是否允许执行 SQL？
```

### 5.1 integrations/llm

建议文件：

```text
model_factory.py
  根据 ModelConfig 创建 chat model。

llm_runtime.py
  聚合 chat_model、callbacks、tags、metadata。

usage.py
  提供 usage metadata 提取和标准化工具。

chains/
  放 LangChain chain 封装。

prompts/
  放 prompt 模板文件。
```

建议定义：

```python
from dataclasses import dataclass
from typing import Any

from langchain_core.language_models import BaseChatModel


@dataclass(frozen=True)
class LlmRuntime:
    chat_model: BaseChatModel
    callbacks: list[Any]
    tags: list[str]
    metadata: dict[str, Any]
```

所有 LLM 调用必须通过这里创建的 runtime 或 chain。

禁止：

```text
业务 service 直接 new chat model。
LangGraph node 直接调用裸模型客户端。
不同 chain 自己随意设置 callbacks / tags / metadata。
```

### 5.2 integrations/database

建议抽象：

```python
from typing import Protocol


class SchemaReader(Protocol):
    def read_schema(self, database_key: str): ...


class SqlExecutor(Protocol):
    def execute(self, database_key: str, sql: str): ...
```

第一版实现：

```text
database/sqlite/schema_reader.py
database/sqlite/executor.py
database/sqlite/initializer.py
```

后续再加：

```text
database/dameng/
database/postgres/
```

### 5.3 integrations/vectorstore

建议抽象：

```text
VectorStore
  search(query, filters, top_k)
  write_documents(documents)
  reset_collection(collection_name)
```

第一版实现：

```text
chroma_store.py
```

业务服务不要直接依赖 Chroma client。

### 5.4 integrations/embedding

embedding 建议单独放：

```text
embedding/factory.py
```

原因：

```text
chat model 和 embedding model 经常 provider / model_name / token usage 规则不同。
```

不要把 embedding 混在 chat model factory 里。

## 6. workflows/runtime 设计

`workflows/runtime` 放 LangGraph 运行层通用能力，不放具体 NL2SQL 业务节点。

建议文件：

```text
graph_runtime.py
  封装 graph invoke / stream。

thread_id.py
  生成和规范化 LangGraph thread_id。
```

### 6.1 graph_runtime.py

职责：

```text
1. 为 graph 调用构造 RunnableConfig。
2. 注入 thread_id。
3. 注入 callbacks / tags / metadata。
4. 提供 invoke / stream 的统一包装。
5. 将 graph 异常转成 WorkflowError。
```

CLI / API 不应该直接拼 LangGraph config。

推荐调用方式：

```python
output = graph_runtime.invoke(
    graph=nl2sql_graph,
    input=workflow_input,
    trace_context=trace_context,
)
```

而不是：

```python
graph.invoke(
    input,
    config={"configurable": {"thread_id": "..."}},
)
```

### 6.2 thread_id.py

职责：

```text
1. 根据 request_id / run_id 生成 thread_id。
2. 保证同一请求可恢复。
3. 避免 CLI / API 各自生成不同格式。
```

## 7. bootstrap 设计

`bootstrap` 是依赖装配层。

建议结构：

```text
bootstrap/
  app.py
  container.py
```

### 7.1 app.py

暴露稳定应用对象：

```python
from dataclasses import dataclass


@dataclass(frozen=True)
class SQLAgentApp:
    nl2sql_workflow: object
    schema_index_workflow: object
    config: object
```

外部入口只依赖 app 暴露的高层能力。

### 7.2 container.py

职责：

```text
1. load config。
2. build logger。
3. build observability。
4. build checkpointer。
5. build LLM runtime。
6. build database gateway。
7. build vectorstore。
8. build services。
9. build workflows。
10. return SQLAgentApp。
```

container 不应该：

```text
写 graph 分支逻辑。
拼 prompt payload。
校验 SQL。
执行 retry 判断。
格式化 CLI 输出。
```

## 8. interfaces 设计

第一版只需要 CLI。

```text
interfaces/cli/main.py
interfaces/cli/commands/ask.py
interfaces/cli/commands/schema_index.py
interfaces/cli/commands/sqlite_init.py
```

CLI 负责：

```text
1. 解析参数。
2. 创建 app。
3. 构造 workflow input。
4. 调用 app 中的 workflow。
5. 格式化输出。
6. 调用 run_recorder 写业务运行记录。
```

CLI 不负责：

```text
schema grounding。
SQL check。
graph edge。
LLM callback。
token 统计细节。
```

## 9. LangGraph、token、日志的关系

### 9.1 LangGraph 负责的观测

LangGraph 可以提供：

```text
node 开始 / 结束
node state update
完整 state values
checkpoint
task error
debug stream
LLM token stream
```

这可以替代旧项目里很多“为了看中间状态而手写日志”的需求。

### 9.2 LangChain / LangSmith 负责的 token usage

token usage 更适合由 LangChain callbacks / LangSmith 处理。

新项目不应该在每个 chain 里手动累加 token。

推荐：

```text
integrations/llm 统一注入 callbacks。
platform/observability 统一读取 usage summary。
LangSmith 开启时交给 LangSmith 记录 trace / token / cost。
```

### 9.3 SQLAgent 自己仍要保留的业务运行记录

LangGraph / LangSmith 不会替你定义 SQLAgent 的业务 run record。

仍然应该自己记录：

```text
run_id
request_id
thread_id
command
question
database_key
status
final_sql
row_count
error
started_at
finished_at
trace_url / trace_id
```

这些放在：

```text
platform/observability/run_recorder.py
```


## 11. 最关键的架构规则

建议写入正式架构文档：

```text
1. 所有 LLM 调用必须经过 integrations/llm。
2. 所有 LangGraph invoke / stream 必须经过 workflows/runtime。
3. 所有 run_id / thread_id / request_id 必须由 platform/observability 生成或规范化。
4. CLI / API 不直接调用 LangGraph graph，只调用 app 暴露的 workflow interface。
5. Node 不直接 new LLM / database / vectorstore，只调用注入的 service。
6. Token usage 和 trace metadata 不在业务 service 里手算。
7. prompt 业务 payload 构建放 services，不放 integrations/llm。
8. container 只负责接线，不判断业务流程。
9. domain 不依赖 LangGraph、LangChain、SQLite、Chroma。
```

## 12. 旧项目痛点如何被新基建解决

旧项目痛点：

```text
1. token 统计散落在 runtime / local runner / chain。
2. chain 日志和 command run record 边界不清。
3. 想看中间状态需要额外写 observation/debug 路线。
4. container 既是装配中心，又容易成为所有依赖的唯一事实来源。
5. CLI 对运行记录、token summary、chain log path 知道太多。
```

新基建对应解决方式：

```text
1. LLM usage 统一从 integrations/llm + platform/observability 走。
2. graph 中间状态优先使用 LangGraph stream/checkpoint。
3. run record 收敛到 platform/observability/run_recorder.py。
4. CLI 只拿 workflow output，不知道内部 chain log path。
5. container 只接线，workflow 逻辑在 workflows。
```

## 13. 当前设计判断

LLM 放在 `integrations/llm` 是更好的结构。

但必须坚持：

```text
LLM 调用基础设施归 integrations。
prompt 业务变量构建归 services。
workflow 节点调用归 workflows。
运行观测归 platform/observability。
依赖接线归 bootstrap。
```

这条边界守住，新项目就不会再重复旧项目“加一点调试功能就牵动全身”的问题。
